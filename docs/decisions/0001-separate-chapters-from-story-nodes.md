# ADR 0001：正文与故事节点分离存储

- 状态：接受
- 日期：2026-06-12

## 背景

原始技术规格把 `chapter_index`、`text_content`、字数和段落范围直接放在
`story_nodes`。StoryNode 同时承担画布位置、时间轴、剧情目标和正文，会使结构编辑、
正文自动保存、版本历史与 AI 生成互相影响。

新增的故事线驱动章节生成还要求分别跟踪：

- 当前剧情节点是否完成。
- 当前故事线里程碑推进程度。
- 章节正文、摘要、字数和生成来源。
- 用户是否接受 AI 的推进建议。

## 决策

采用独立 `chapters` 表：

- `story_nodes` 只保存剧情结构、排序、时间、画布位置和生成目标。
- `chapters` 保存章节序号、标题、正文、摘要、字数、来源和修订号。
- MVP 中 `chapters.story_node_id` 唯一，即一个 StoryNode 至多对应一章。
- StoryNode 可以先存在而没有正文，适合规划阶段。
- 章节可独立修订，不改变画布或时间轴结构。

故事线阶段目标使用独立 `storyline_milestones` 表，不用 JSON 数组：

- `sort_order` 定义推进顺序。
- `status` 与 `progress` 记录推进状态。
- 当前里程碑由第一个未完成里程碑推导，避免额外指针失效。
- AI 推进报告在用户确认后才更新里程碑。

## 后果

优点：

- 正文编辑、版本历史和结构画布边界清晰。
- 可通过外键与唯一约束保证 MVP 的一节点一章。
- 后期移除唯一约束即可支持一节点多章。
- 故事线生成可以查询当前节点、相邻节点、剩余里程碑和结局目标。

代价：

- 读取“节点 + 正文”需要一次关联查询。
- 节点排序和章节编号是两个概念，需要服务层保持业务一致性。
